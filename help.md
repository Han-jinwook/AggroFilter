# 📘 전략 문서: Hydration 안정화 + 분석 API 2분리 실행계획
**작성일**: 2026-04-26 14:00 (KST)
**문서 목적**: 실패 이력이 아닌, 앞으로의 실행 전략과 단계별 계획을 명확히 고정

---

## 1) 전략의 취지
현재 우선순위는 기능 확장이 아니라, 렌더 안정성과 분석 흐름 분리를 통해 서비스의 기본 동작을 견고하게 만드는 것이다.

핵심 취지:
- SSR/CSR 초기 렌더 불일치를 구조적으로 줄인다.
- 로그인/모달/헤더/설정 페이지의 렌더 책임을 명확히 분리한다.
- 분석 요청과 결과 반환을 분리해 UX 지연과 렌더 충돌을 동시에 완화한다.

---

## 2) 전략의 목적

### 제품 목적
1. 로그인 이후 주요 페이지 이동에서 사용자 체감 오류를 제거한다.
2. 분석 요청 직후 즉시 피드백을 제공해 "멈춘 화면" 인상을 없앤다.
3. 안정화 이후 비동기 UX 확장을 위한 기반을 만든다.

### 기술 목적
1. 최초 렌더 트리를 서버/클라이언트에서 동일하게 유지한다.
2. 브라우저 의존 상태 접근을 effect 단계로 제한한다.
3. 분석 처리 파이프라인을 `request`/`result`로 분리해 책임을 명확히 한다.

---

## 3) 고정 원칙 (변경 금지)
1. 안정화 완료 전 기능 추가 금지
2. 로그인 모달 구조 단일화 (전역 1개 또는 페이지별 1개 중 택1)
3. 렌더 중 `window`/`localStorage` 분기 금지
4. mounted 가드 최소 적용 (페이지 전체 가림 방식 금지)
5. 검증 순서 고정: `/` → `/p-settings` → `/p-notification`
6. 변경 단위 최소화: 한 번에 한 원인군만 수정

---

## 4) 아키텍처 전략: 분석 API 2분리

### A. Request API (`POST /api/analysis/request`)
역할:
- 요청 접수
- 최소 입력 검증
- 분석 작업 ID 발급
- 즉시 응답

응답 예시 필드:
- `analysisId`
- `status` (`queued` | `processing`)
- `acceptedAt`

### B. Result API (`GET /api/analysis/result/[id]`)
역할:
- 작업 상태 조회(폴링)
- 부분 결과/최종 결과 반환
- 실패 시 에러 상태와 사유 전달

응답 예시 필드:
- `status` (`queued` | `processing` | `completed` | `failed`)
- `progress` (선택)
- `result` (완료 시)
- `error` (실패 시)

### 기대 효과
- 초기 렌더와 무거운 분석 처리를 시간적으로 분리
- 로그인 직후 라우팅/모달 상태와 분석 처리 간 결합도 축소
- Hydration 리스크 구간의 복잡도 감소

## 5) 단계별 실행 계획

### Phase 0. 사전 고정
목표:
- 기준 브랜치/커밋 상태 확정
- 작업 범위와 금지 범위 명시

작업:
1. 모달 소유권 모델 확정 (전역/페이지 중 1개)
2. 점검 대상 파일 확정 (`AppHeader`, `p-settings`, `p-notification`, 분석 요청/결과 API)
3. 검증 시나리오 체크리스트 파일화

완료 기준:
- 수정 대상/비대상 합의 완료

### Phase 1. 렌더 안정화 (Hydration 1차)
목표:
- 로그인 직후 `/p-settings`까지 hydration error 0

작업:
1. 렌더 경계 점검표 작성 (SSR 출력 vs CSR 초기 출력)
2. 렌더 단계 브라우저 접근 제거
3. 로그인 상태 파생값 계산 시점 통일(effect 이후)
4. 모달 렌더 경로 단일화

완료 기준:
- `/` 진입, 로그인, `/p-settings` 이동, 새로고침 반복에서 콘솔 hydration 에러 0

### Phase 2. API 2분리 적용
목표:
- 분석 흐름을 request/result 책임으로 분리

작업:
1. `request` API는 "접수+ID 발급"까지만 수행
2. `result` API는 상태 조회/결과 반환만 수행
3. 프론트는 `analysisId` 기반 폴링 상태머신 적용
4. 로딩/부분완료/완료/실패 UI 상태 분리

완료 기준:
- 분석 요청 직후 즉시 응답
- 폴링으로 정상 완료/실패 전이
- 렌더 불일치 없이 결과 화면 전환

### Phase 3. 페이지 확장 검증
목표:
- `/p-notification`까지 동일 안정성 확보

작업:
1. 알림 페이지의 인증 파생값 초기화 시점 통일
2. 헤더/알림 배지 렌더 순서 점검
3. 로그인 상태 전환(로그인/로그아웃/새로고침) 반복 검증

완료 기준:
- 고정 시나리오 전 구간 콘솔 에러 0

### Phase 4. 운영 안정화
목표:
- 재발 방지 장치 마련

작업:
1. PR 템플릿에 "SSR/CSR 분기 체크" 항목 추가
2. 주요 페이지 수동 회귀 시나리오 문서화
3. 분석 API 계약(JSON 필드) 고정

완료 기준:
- 동일 이슈 재발 시 탐지/복구 시간이 단축되는 운영 기준 확보
