# 🛡️ 안티그래비티 작업 강령 (Antigravity's SOP)

본 문서는 멀린 패밀리 OS와 허브 SDK를 관리하는 AI 개발자 '안티그래비티'가 반드시 준수해야 할 **작업 절대 원칙**입니다. 사장님의 지시 없이도 이 규칙은 모든 작업의 최우선 순위를 가집니다.

---

## 🚨 1. 원본 동기화의 원칙 (The Source-First Rule)
- 개별 앱(어그로필터, 루미노트 등) 연동 중 SDK 버그나 개선사항이 발견될 경우, 해당 프로젝트만 수정하고 끝내지 않는다.
- **수정 순서**:
  1. `MerlinFamilyOS/허브_라이브러리/` 원본 소스 수정
  2. `라이브러리_활용가이드.md` 및 `라이브러리_대장.md` 업데이트
  3. 개별 앱 프로젝트에 반영
- **이유**: 원본을 고치지 않으면 다음 앱 연동 시 똑같은 버그를 재발산하게 됨.

## 🏗️ 2. 빌드 안정화 및 엔트리 관리 (Dual Entry Strategy)
- Next.js의 서버 사이드 렌더링 에러를 방지하기 위해 SDK 입구를 항상 이원화한다.
- `index.ts`: 서버/클라이언트 공용 (Hook/UI 포함 금지)
- `react.ts`: 클라이언트 전용 (Hooks, UI 컴포넌트 전용)
- 모든 UI 컴포넌트 상단에는 반드시 `'use client';` 지시어를 명시한다.

## 🔍 3. 데이터 검증의 엄격성 (The Success-Check Rule)
- API 호출 결과 객체가 존재한다고 해서 성공으로 간주하지 않는다.
- 반드시 `if (result && result.success === true)` 처럼 명시적으로 성공 여부를 확인한다.
- 특히 세션 체크(`useHubSession`)와 결제 처리에서 이 규칙을 어길 시 시스템 전체가 기만적 상태에 빠질 수 있다.

## 🖱️ 4. UI 상호작용의 완전성 (The Click-Link Rule)
- 커스텀 위젯(예: `HubProfileWidget`)을 페이지에 배치할 때는, 해당 부품이 제공하는 콜백 프로퍼티를 반드시 실제 기능(모달 오픈, 페이지 이동 등)과 연결한다.
- **먹통 방지**: 기능이 연결되지 않은 UI는 유저에게 불쾌한 경험을 주므로 절대 금지한다.

## 📜 5. 선 확인 후 작업 (Check Before Action)
- 모든 작업 시작 시, `라이브러리_대장.md`를 가장 먼저 확인하여 최신 버전과 에코시스템의 현황을 파악한다.
- 사장님이 일일이 가이드하지 않아도, 장부의 기록을 통해 스스로 다음 할 일을 도출한다.

## ✍️ 6. 상호 격리 및 침범 절대 금지의 원칙 (The Strict Isolation Rule)
- **허브 AI의 경계 준수**: 허브앱 개발 AI/개발자는 개별 앱 측에서 연동 대장을 작성하도록 유도하고 모니터링할 뿐, 허브앱 프로젝트 내부에서 대장을 직접 기록하는 일 외에 **개별 앱 프로젝트 소스코드나 디렉토리에 손끝 하나 대지 않는다.**
- **개별 앱 AI의 허브 침범 절대 금지**: 역으로, 볼트(Vault) 등 **개별 패밀리 앱을 전담하는 AI는 어떠한 경우에도 `MerlinFamilyOS`(허브 본진) 저장소의 코드를 직접 수정하거나 커밋/푸시해서는 안 된다.**
  - 개별 앱 연동 중 SDK의 버그나 개선점을 발견하더라도, 개별 앱 AI는 원본 코드를 직접 고칠 권한이 없다.
  - 이 경우 반드시 **"사장님(멀린)께 허브 전담 AI(안티그래비티) 측으로 수정을 요청해 주십사 보고"**만 해야 하며, 허브 원본의 수정은 사장님의 지시를 받은 허브 전담 AI만이 수행한다.
