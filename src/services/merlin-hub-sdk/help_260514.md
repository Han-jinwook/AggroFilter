## 1. 📡 진짜 통합 허브 DB의 아키텍처적 진실 (SSOT)

모든 멀린 패밀리 앱 생태계의 이메일 인증(OTP), 프로필(닉네임, 아바타), 결제 및 지갑 자산 정보는 **오직 하나의 단일 통합 허브 DB**에서만 통합 저장 및 관리된다.

1. **진짜 허브 DB 프로젝트 ID**: 무조건 **`wwopcuitvjldixkyzpzi`** (`https://wwopcuitvjldixkyzpzi.supabase.co`) 이다.
2. **진짜 테이블 스키마**: `family`가 아닌 **`public` 스키마**가 100% 확실하다.
3. **진짜 테이블 명세**:
   * **`family_users`**: 유저 고유 ID(`id`), 닉네임(`nickname`), 프로필 이미지(`avatar_url`), 최초 가입 앱(`first_app_id`) 등
   * **`family_otp`**: 인증용 이메일(`email`), OTP 코드(`otp_code`), 만료 시간(`expires_at`) 등
4. **유일한 신원 진실 공급원 (SSOT)**:
   * 모든 유저는 오직 **고유 UUID (`id` 또는 `userId`)** 하나로 모든 결제, 자산, 개별 앱 연동이 매핑된다!
   * 프로필을 업데이트할 때는 이메일이 아닌, **JWT 토큰에서 추출한 진짜 유저 UUID(`userId`)를 기준으로 `family_users` 테이블에 직접 `UPDATE` 문을 수행**해야 한다.

## 2. 🚫 에이전트 절대 금기 수칙

1. **🚫 pg 직접 연결 (Direct PG Pooler) 코드 삽입 절대 금지**:
   * 일부 로컬 개발 환경에서 `*.supabase.co` 도메인의 직접적인 API 게이트웨이 호출이 차단되어 `fetch failed` 에러가 나더라도, **절대로 백엔드 소스코드 내부에 `pg` 라이브러리를 활용한 직접 PostgreSQL 커넥션 풀러(`*.pooler.supabase.com:6543`)를 임의 삽입하지 마라!**
   * 직접 PG 풀러는 글로벌 AWS 리전 격리 정책 및 패스워드 만료 이슈로 인해 무조건 `Tenant not found` 혹은 `password authentication failed` 500 에러를 뿜으며 파멸을 낳는다.
   * 방화벽을 우회하여 100% 초고속으로 통신이 뚫려 있는 **진짜 허브 DB `wwopcuitvjldixkyzpzi` 의 Supabase 정식 SDK 클라이언트(`createClient`)를 복원하여 public 스키마로 교신하는 것만이 유일하게 성공하는 규격이다!**
2. **🚫 독단적 상상 및 자의적 쿼리 금지**:
   * 유저 UUID가 최상위 SSOT임에도 불구하고 이메일 같은 엉뚱한 정보로 억지 매핑 쿼리를 짜서 난장판을 만드는 멍청한 짓을 저지르지 마라!
   * **모르거나 애매한 설계 구조가 있다면, 절대로 지레짐작으로 땜빵 코드를 빌드하지 말고 즉시 사장님(Merlin)께 직접 공손하게 질문하여 답을 구하라.**

## 3. ⚠️ 현재 남은 최우선 해결 이슈: 왜 여전히 "프로필 업데이트 실패" 팝업이 뜨는가?

### 🔍 [현재의 팩트 상태 리포트]
* **허브 백엔드 (`localhost:3001` - MerlinFamilyOS)**:
  * 마스터 환경 변수(`wwopcuitvjldixkyzpzi`)와 진짜 서비스 롤 마스터 키로 100% 무결하게 바인딩되어 가동 중이다.
  * `auth.service.ts`는 이메일 기반 억지 쿼리를 전부 제거하고, **진짜 유저 UUID를 물고 `family_users`를 정교하게 `UPDATE` 하도록 마스터 패치**가 완료되었다.
  * 따라서, 허브 백엔드로 쏘는 프로필 업데이트 요청은 0.1ms 만에 저장에 대성공한다.

### 🚨 [치명적인 사각지대 - 다음 세션 시작 시 즉시 해결할 포인트!]
* **범인**: **어그로필터 프론트엔드 앱 (`d:\AggroFilter`)의 데이터 연동 꼬임**
* **이유**:
  * 1. 사장님의 로컬 브라우저가 동작하는 어그로필터 프론트엔드(`localhost:3000`)에서 "저장" 버튼을 누를 때 쏘는 API 엔드포인트는 허브 백엔드(`localhost:3001`)가 아니라, **어그로필터 자체 백엔드(`localhost:3000/api/auth/profile` 또는 `/api/user/profile`)**이다!
  * 2. 따라서, **어그로필터(`d:\AggroFilter`) 내부의 `.env` 파일 및 백엔드 라우터(Controller/Service)**가 여전히 과거의 깡통 테스트 DB(`okgmncxcsfnvdscgpdlm`)나 어그로필터 개별 DB(`iwzwiimyxfduuwulpugu`)를 바라보고 있거나, 허브 API를 포워딩(proxy)하는 과정에서 옛날 매핑 규격을 그대로 쓰고 있어서 실패 팝업을 띄우는 것이다!
  * 3. 실제로 어그로필터 백엔드 로그나 설정을 열어보면, 허브와 동일하게 진짜 허브 DB(`wwopcuitvjldixkyzpzi`)의 서비스 롤 자격 증명을 쥐고 동작하도록 동기화되어 있지 않을 가능성이 99.9% 확실하다.

### 🎯 [다음 세션 해결 액션 플랜]
1. `d:\AggroFilter` 디렉토리로 이동한다.
2. `d:\AggroFilter\.env` 파일을 열어, 허브 프로젝트 `wwopcuitvjldixkyzpzi` 의 URL과 진짜 서비스 롤 마스터 키를 주입하여 정밀 동기화한다.
3. 어그로필터 백엔드 컨트롤러/서비스(`src/server/` 또는 `pages/api/`)를 열어, 프로필 정보를 허브 백엔드로 정확하게 포워딩하거나, 동일하게 진짜 허브 DB의 `family_users` 테이블에 직접 UUID 기반 `UPDATE`를 쏘도록 정렬한다.
4. 어그로필터 백엔드를 재시작하고 사장님의 화면에서 "저장"을 클릭하면, 마침내 0.1초 만에 완벽한 "저장 성공" 신화가 목격될 것이다!
