## 7. 이메일 알림 리다이렉션 설정 및 동적 발송 응용 가이드

허브의 `sendNotification`을 이용하여 이메일을 발송할 때, 메일 내에 포함되는 링크 버튼의 URL(`link`, `link2`)을 설정할 때 **절대 경로**를 입력해야 이메일 환경에서 정상적으로 클릭이 가능합니다. 

### 7.1 환경변수 기반 동적 URL 설정
Netlify, Vercel 등의 환경에서 동작할 때와 로컬 환경을 모두 지원하기 위해, 코드 내부에서 다음과 같이 동적 Base URL을 생성하여 사용하는 것을 권장합니다.
```typescript
// Next.js (Netlify 또는 Vercel 환경)
const baseUrl = process.env.NEXT_PUBLIC_SITE_URL || process.env.URL || 'http://localhost:3000';

await hubClient.sendNotification({
  userId: actualUserId,
  title: `[알림] 분석 완료`,
  content: '분석이 완료되었습니다.',
  link: `${baseUrl}/result/123`,
  link_text: '자세히 보기'
});
```

### 7.2 결제 영수증 통합 이메일 발송 응용
기본적으로 허브 서버는 결제 및 코인 차감(`chargeDynamic`, `useCredit`, `processTransaction`) 이벤트 발생 시 자체적으로 코인 차감 영수증 메일을 전송합니다.
하지만 서비스에서 발생하는 특정 이벤트 알림과 코인 차감 안내를 **하나의 메일로 통합**하고 싶다면, 과금 시 `skipReceiptEmail: true`를 전달하여 허브의 자동 발송을 차단한 후, `sub_content_html`에 영수증 HTML을 추가하여 발송할 수 있습니다.

```typescript
// 1. 과금 호출 (영수증 자동 발송 스킵)
await hubClient.chargeDynamic({
  userId,
  videoId,
  // ...
  skipReceiptEmail: true 
});

// 2. 통합 이메일 발송
const receiptHtml = `
  <div style="background-color: #f8fafc; border: 1px solid #e2e8f0; border-radius: 12px; padding: 20px; margin-bottom: 30px;">
    <!-- 영수증 테이블 HTML (코인 차감 내역 등) -->
  </div>
`;

await hubClient.sendNotification({
  userId,
  title: '[서비스 알림] 작업 완료',
  content: '요청하신 작업이 처리되었으며, 서비스 이용에 코인이 사용되었습니다.',
  sub_content_html: receiptHtml, // 본문 아래에 영수증 테이블이 삽입됨
  link: `${baseUrl}/result/123`,
  link_text: '결과 보기',
  link2: `${baseUrl}/payment/purchase?tab=history`,
  link2_text: '이용 내역 확인'
});
```
이러한 조합을 사용하면 사용자가 불필요한 메일을 여러 번 받지 않고 하나의 정돈된 메일로 알림과 영수증을 확인할 수 있습니다.

## 🎨 2. UX/UI 템플릿 표준 규격

- **사이드 윙 배치 공식 및 간격 (CSS 변수 기준)**:
  - **위치 공식**: 
    - `left: calc(50% + (var(--app-max-width) * 0.5 + var(--app-wing-gutter)))`
    - `top: calc(var(--app-header-height, 80px) + 16px)` (각 앱마다 헤더 높이가 다르므로 `--app-header-height` 변수를 기반으로 동적 16px 여백 확보)
  - **관련 규격**:
    - `--app-max-width`: `800px` (본문 최대 가로폭)
    - `--app-header-height`: 각 앱에 정의된 헤더 높이 픽셀 (예: 어그로필터 80px, 타 앱 72px 등 가변 적용)
    - `--app-wing-gutter`: `6px` (본문 박스 끝선 ~ 사이드 윙 사이의 마진)
    - `--app-wing-width`: `160px` (사이드 윙 가로폭)
  - **시각적 실질 간격**: 본문 박스 내부에 걸린 패딩(`px-4` = 16px)과 윙 간격(`6px`)이 합산되어, 화면 상에 렌더링된 본문 카드 우측선과 공유 카드 좌측선 사이의 실제 시각적 여백은 **`22px`**가 됩니다.
- **반응형 하단 광고 전환**: 화면 폭이 좁아져 우측 광고가 사라질 때는, 하단 앵커(Sticky Bottom) 등으로 대체합니다.

### 2.2 전 페이지 레이아웃 가로폭 및 패딩 표준 규격
패밀리 앱 전체의 통일감 있는 고품격 UX 구현을 위해 아래 **가로폭 수치(Pixel)**를 정확히 준수하여 모든 페이지와 카드 컴포넌트를 설계해야 합니다.

* **헤더 및 모든 페이지 최대 가로폭 (공통)**: **`800px`** (`max-w-[800px]` 또는 `max-w-[var(--app-max-width)]`)
  * 헤더(AppHeader)와 모든 페이지 메인 콘텐츠 영역의 가로폭은 **무조건 `800px`로 동일하게 제한**하여 좌우 정렬을 완전히 일치시킵니다.
  * **컨테이너 좌우 패딩 통일 (필수)**: 헤더 내부 컨테이너와 모든 페이지 메인 콘텐츠 컨테이너의 좌우 패딩은 모바일 및 데스크톱 환경 전체에서 동일하게 **`px-4`** (16px)로 완전히 일치시킵니다. 본페이지에 `lg:px-8`과 같은 차등 패딩을 선언할 경우, 데스크톱 환경에서 본문 영역만 안쪽으로 좁아져 정렬선이 깨지므로 절대 이를 금지합니다.
  * **헤더와 본문 세로 간격 표준화 (필수)**: 가변적인 헤더 높이에 대응하여 헤더 하단 경계선과 본문 카드 시작점 사이의 간격은 항상 **`16px`**로 통일합니다. 본문 `main` 컨테이너 상단에 `pt-4` (16px) 혹은 헤더 고정식의 경우 `pt-[calc(var(--app-header-height)+16px)]`를 주어 여백을 칼같이 일치시켜야 합니다.
  * **🚨 헤더 고정 및 Glassmorphism 적용 규칙 (추천)**:
    * 스크롤 시에도 헤더가 상시 노출되도록 `fixed top-0 left-0 right-0 z-50` (또는 `sticky top-0 z-50`) 클래스 사용을 표준으로 규정합니다.
    * 본문 카드가 헤더 하단으로 부드럽게 흘러 비쳐 보일 수 있게 헤더 배경은 불투명 대신 **반투명화 및 흐림 효과(`bg-white/70 backdrop-blur-md` 등 정확히 70% 수준의 알파 투명도 적용)**를 공식 권장합니다.

### 2.3 헤더 메뉴 및 UI 텍스트 언어 표준 (통일성)
패밀리 앱들 간의 이질감을 없애고 통일된 브랜드 보이스를 전달하기 위해 아래 **메뉴명 네이밍 규칙 및 배치 규칙**을 엄격히 준수합니다.

* **100% 한글화 원칙**: "My Page", "Admin", "분석 Plaza" 등 영어나 혼용된 표현의 사용을 절대 금지합니다.
* **표준 용어 사전**:
  * `My Page` ➡️ **`마이페이지`** (공백 없음)
  * `Admin` ➡️ **`관리자`**
  * `분석 Plaza` ➡️ **`분석 플라자`**
* 버튼 텍스트나 메뉴 명칭은 직관적이고 친근한 **순수 한글**을 우선적으로 사용해야 합니다.
* **헤더 메뉴 및 프로필 반응형 배치 규칙**:
  * **모바일 화면**: 공간 제약으로 인해 메뉴는 **아이콘을 숨기고 텍스트(메뉴명)만** 노출하며, 프로필 위젯은 반대로 **닉네임을 숨기고 아바타 이미지만** 노출합니다. (`hidden sm:flex` 및 `hidden sm:block` 활용)
  * **PC 화면**: 공간이 충분하므로 메뉴는 **[아이콘 + 메뉴명]** 형태로 모두 표시하며 **반드시 나란히 가로 배치(`sm:flex-row`)** 합니다. (단, 프로필 위젯과 코인 버튼은 아바타/금액 아래 텍스트가 오는 기존 세로 배치(`flex-col`) 구조를 유지합니다.)

* **레이아웃 그리드 표준 규격 (2컬럼 고정)**:
  * 본페이지는 무조건 **2컬럼 그리드** (`grid grid-cols-1 lg:grid-cols-2 gap-8`)로만 구성하여 메인 카드들이 각각 **`384px`**의 가로폭을 정상적으로 확보하도록 합니다.
  * 단일 컬럼 렌더링 시(로그아웃 등)에는 최대 가로폭 **`576px`** (`max-w-xl`)로 제한해 중앙 정렬합니다.
* **공유 카드 (`HubShareSquare`) 및 사이드 윙 복합 배치 절대 규칙**:
  * **본페이지 내부(800px 그리드 안) 배치 절대 금지**: 공유 카드는 본페이지 내부에 절대 임베딩하지 않습니다.
  * **우측 외부 고정 및 absolute 격리 배치 (필수)**: 
    * 공유 카드는 항상 본페이지 레이아웃 외부의 **우측 영역(사이드 윙)**에 독립적으로 고정 플로팅하여 배치해야 합니다.
    * **배너 레이아웃 간섭 금지**: 배너나 윙 카드가 메인 본문 컨테이너의 flex 흐름을 해치고 수평 정중앙 정렬을 밀어내어 틀어지게 만드는 행위를 방지하기 위해, 우측 `aside`는 반드시 **`absolute` 포지셔닝으로 격리 배치**합니다.
    * **배치 위치 공식**: `absolute top-[var(--app-header-height)] left-[calc(50%+400px+8px)] h-[calc(100vh-var(--app-header-height))]` (본문 너비 800px의 절반선 기준 우측 8px 외곽선 격리 영역 배치)
  * **공유 카드 + 광고 배너 복합 스택 규격 (겹침 방지)**:
    * 공유 카드와 광고 배너가 우측 사이드 윙에 세로로 공존할 경우, 개별 카드에 각각 `sticky`를 주면 스크롤 시 카드가 겹치는 문제가 발생하므로 **두 카드를 감싸는 부모 Wrapper 컨테이너에 `sticky top-[calc(var(--app-header-height)+16px)]`을 적용**해야 합니다.
    * **가로폭 균일화 (160px 통일)**: 두 카드의 가로 정렬선이 일치하도록 광고 배너 및 공유 카드의 너비를 `w-full`로 통일하여 부모 `aside` 너비(`w-[160px]`)를 꽉 채우도록 구성합니다.
    * **마우스 호버 커서 피드백 (전역 규칙)**: 프로필 내부 요소(설정, 취소, 저장, 로그아웃 버튼 등), 헤더 메뉴, 링크 복사 버튼을 포함하여 유저가 클릭/터치할 수 있는 **모든 인터랙티브 요소**에는 마우스 호버 시 반드시 손가락 모양 커서가 나오도록 `cursor-pointer` 유틸리티 클래스가 누락 없이 명시적으로 지정되어 있어야 합니다. (SDK v1.1.1 및 전체 전수조사 수정 완료)
  * **🚨 공유 카드 노출 페이지 적용 범위 (필수)**:
    * 공유 카드는 특정 서브 페이지(설정, 프로필 등)에만 제한적으로 노출하지 않습니다.
    * 사용자의 앱 유입 및 바이럴 극대화를 위해, **앱의 기본 메인 홈 화면(Landing/Main View) 및 주요 콘텐츠 뷰 전체에 상시 노출**될 수 있도록 공통 레이아웃 또는 루트 컨테이너 레벨에서 일괄 렌더링해야 합니다.
  * **🚨 공유 대상 페이지 강제 우회 정책 (필수 - SDK v1.0.2 내장)**:
    * 비로그인 피초대자가 `/profile`이나 `/admin7878` 등 로그인/인증이 강제되는 서브 페이지에 즉시 유입되는 경우, 첫 진입 직후 뜬금없이 로그인 팝업이 전면 강제 표시되어 이탈율이 극대화되는 파멸적 UX가 발생합니다.
    * 이를 방지하기 위해 공유 카드(`HubShareSquare`)는 내장된 `privatePaths` 필터를 통해 `/profile`, `/whateat/profile`, `/admin7878` 등의 전용 내부 주소 링크 복사 시, 이를 자동으로 루트(`/`, 온보딩 페이지) 주소로 변경하여 초대 코드(`?ref=초대코드`)와 결합시킵니다.
    * 피초대자는 온보딩 화면을 편안히 둘러본 뒤, 유저 자발적 액션을 통해 가입을 결정하게 됩니다.
* **🚨 PWA 및 브라우저/마우스 뒤로가기 내비게이션 표준 (필수)**:
  * **목적**: 모바일 PWA Standalone 모드 및 PC 웹 브라우저 환경(마우스 뒤로가기 핫키 포함)에서 뒤로가기 액션 시 앱이 통째로 종료되거나 초기 화면으로 강제 리셋되는 UX 파멸 현상을 방지합니다.
  * **동기화 원칙**:
    1. **히스토리 적재 (pushState)**: 탭 전환 또는 모달 활성화 시 `history.pushState({ tab: targetTab }, '', '#hash')` 형식을 통해 브라우저 세션 스택에 페이지 상태를 적재해야 합니다. 이때 무한 히스토리 스택 루프 방지를 위해 `window.history.state` 검사 등의 멱등성 가드를 필수 적용합니다.
    2. **popstate 이벤트 리스닝**: 컴포넌트 마운트 시 `popstate` 이벤트를 등록하여 브라우저 뒤로가기/앞으로가기 액션 시 `event.state`에 매핑된 탭 상태를 추적 및 복원하여 뷰(View)를 강제 동기화합니다.
    3. **제스처 오버스크롤 제어**: iOS 스와이프 뒤로가기 시의 화면 들썩거림을 억제하기 위해 레이아웃 바디 또는 메인 컨테이너에 `overscroll-behavior-x: contain` 적용을 권장합니다.

## 🎨 3. 비회원 온보딩 카드 & 메뉴 동적 하이라이트 표준 규격

비회원 진입 시 서비스의 핵심 가치와 기능을 한눈에 인지시키고 가입(Sign-up) 전환율을 극대화하기 위한 대화형 온보딩 표준 템플릿 규격입니다.

### 3.1 핵심 동작 원리 및 스펙
1. **메뉴 매핑 카드 그리드 (2~4개 구성)**:
   - 각 앱의 핵심 메뉴/기능을 요약한 카드를 그리드 형태(`grid-cols-2` 등)로 배치합니다.
2. **동적 헤더 메뉴 하이라이트 (Hover & Focus Sync)**:
   - 유저가 특정 기능 카드에 마우스 호버(`onMouseEnter`) 시, **상단 헤더(Header)의 내비게이션 탭이 해당 기능의 탭 ID로 동적으로 변경(하이라이트)**되어야 합니다. 마우스가 벗어날 시(`onMouseLeave`)에는 원래 탭 상태로 복구됩니다.
   - 이를 통해 유저가 클릭 후 진입하게 될 기능의 위치를 시각적으로 직접 인지시켜 사용성을 극대화합니다.
3. **하단 가입 전환 CTA 버튼 (3초 가입 Nudge)**:
   - 카드 영역 하단에는 눈에 잘 띄는 대형 버튼(Call-to-Action)을 배치합니다.
   - **미로그인 상태**: `"3초만에 시작하기 ⚡"` 또는 `"이메일로 3초 로그인하기"` 문구를 노출하며, 클릭 시 허브 로그인 모달(`openLoginModal` 이벤트)을 트리거합니다.
   - **로그인 상태**: `"서비스 시작하기"` 또는 해당 앱의 핵심 액션 문구를 노출하며 바로 메인 화면으로 이동합니다.
4. **앱별 시그니처 브랜딩 컬러 커스텀 (디자인 자유도)**:
   - 온보딩 템플릿의 인터랙션 흐름(호버 연동, CTA 동작)은 통일하되, **각 앱의 고유 시그니처 색상(브랜드 메인 컬러)에 맞추어 카드 호버 보더, 데코레이션 그라데이션, 아이콘 배경 및 버튼 배경색 등은 각 서비스의 아이덴티티에 알맞게 자유롭게 커스텀하여 적용**합니다. (예: 뭐먹지는 따뜻한 오렌지/핑크 톤을 사용하고, 분석 전문 앱은 신뢰감을 주는 블루/바이올렛 계열을 사용하는 등 자율적인 스타일 지정을 적극 권장합니다.)

### 3.2 표준 리액트 온보딩 템플릿 코드

```tsx
import { sendNotification } from '@/services/merlin-hub-sdk/CoreLogic/client';
```
