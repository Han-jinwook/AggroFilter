본 문서는 Merlin Hub SDK의 기술적인 사용 방법과 모든 모듈별 예제 코드를 관리합니다.

## 🛠️ 0. 기본 설정 (Setup)

1. **에셋 복사**: `허브_라이브러리/assets/`의 파일들을 개별 앱의 `public/hub_assets/` 폴더로 복사합니다.
2. **Core 라이브러리 연동**: `허브_라이브러리` 내의 모든 폴더(Auth, Session, Wallet 등)와 파일을 앱의 `src/services/merlin-hub-sdk/` 경로에 배치합니다.

### 2단계: 에셋(이미지) 수동 배치 (필수)
부품들이 사용하는 공식 이미지들을 브라우저가 인식할 수 있도록 배치합니다.
1. 개별 앱의 `public/` 폴더 내에 `hub_assets/` 폴더를 생성합니다.
2. `src/services/merlin-hub-sdk/assets/` 내의 모든 파일을 `public/hub_assets/`로 복사합니다.
   - 예: `guest_paw.png` → `public/hub_assets/guest_paw.png`

### 3단계: 통합 배선 및 레이아웃 설정
1. **입구 이원화 (필수)**: 
   - **`@/src/services/merlin-hub-sdk`**: API Route, Server Component용 (Pure TS)
   - **`@/src/services/merlin-hub-sdk/react`**: UI, Hooks용 ('use client' 선언됨)
2. **알림 연동**: `app/layout.tsx` (또는 최상위 레이아웃)에 `<HubNotifier />`를 배치하여 시스템 전체 알림을 활성화합니다.

> **🚀 [중요] SPA (Vite, CRA 등) 환경 연동 시 주의사항 및 설계 철학**
> - **프레임워크 비종속성 (Agnostic SDK)**: v1.2.2 부터 `next/navigation` 등 특정 프레임워크 전용 패키지에 의존하지 않고 순수 Web API(`window.location`)를 사용하도록 개선되었습니다.
>   - **철학**: *"Next.js용 모듈과 React용 모듈을 분리하면 유지보수 비용과 버그 발생 확률이 2배가 됩니다."* 단일 표준 Web API 코드로 유지하여 파편화를 막고 완벽한 호환성을 보장하는 것이 핵심 설계 원칙(Single Source of Truth)입니다.
> - **`'use client'` 지시어**: SDK 파일 상단의 `'use client'`는 Next.js 환경과의 호환성을 위한 것으로, Vite/CRA 같은 순수 React 환경에서는 **안전하게 무시**됩니다. 에러가 나지 않으므로 임의로 지울 필요가 없습니다.
> - ⚠️ **[AI 개발자 필수 숙지] API 통신 프록시(Proxy) 브릿지 강제 규정**
>   개별 앱의 프론트엔드 브라우저에서 허브 백엔드(`os.sundreamer.app`)로 **직접(Direct) API를 호출하는 것은 절대 금지**됩니다. 직접 호출 시 브라우저의 CORS 차단 및 애플(Safari) ITP(서드파티 쿠키 차단) 정책에 의해 인증 세션이 모두 유실됩니다.
>   **반드시 개별 앱 자체의 서버/엣지를 경유하도록 다음과 같이 우회로(Proxy)를 뚫고 SDK의 `hubUrl`을 상대경로(`""`)로 설정하여야 합니다.**
> 
>   *(1) 로컬 개발 환경 (`vite.config.js`)*
>   ```javascript
>   export default defineConfig({
>     server: {
>       proxy: {
>         '/api': {
>           target: 'https://os.sundreamer.app',
>           changeOrigin: true,
>           secure: false,
>         }
>       }
>     }
>   });
>   ```
>   *(2) 프로덕션 배포 환경 (`netlify.toml` 예시)*
>   ```toml
>   [[redirects]]
>     from = "/api/*"
>     to = "https://os.sundreamer.app/api/:splat"
>     status = 200
>     force = true
>   ```
>   💡 **[중요 트러블슈팅] Netlify 배포 시 API가 404 또는 HTML을 반환하는 경우**
>   만약 개별 앱에 빌드 시점(Build time)에 `_redirects` 파일을 동적으로 생성하는 스크립트(예: `write-dist-redirects.mjs` 등)가 존재한다면, Netlify의 정책상 동적 생성된 `dist/_redirects` 파일이 최상단 `netlify.toml`의 설정보다 **우선순위가 높아 프록시 규칙이 무시**됩니다.
>   이 경우 `netlify.toml`에 적는 대신, 해당 `_redirects` 생성 스크립트 내부에 직접 프록시 규칙(`/api/*  https://os.sundreamer.app/api/:splat  200!`)을 하드코딩으로 추가해주어야만 SPA 폴백(Fallback)에 먹히지 않고 정상 중계됩니다.

> **📝 [메모] 타 앱 교차 진입 시 (자동 로그인) 가입 축하 코인 정책**
> 허브 생태계 내 다른 앱으로 넘어갈 경우, 해당 앱에서 '최초 기능 사용(코인 소모 액션)'을 할 때 이를 가입으로 간주하여 "해당 앱 가입 축하 500C"를 다시 지급해야 합니다.
> 단, 코인을 사용하지 않는 앱(예: 썬드리머)의 경우 축하 500C 지급 로직은 없습니다. (각 개별 앱의 서비스 로직에서 이 최초 액션 트리거를 구현해야 합니다.)

## 🧩 1. 프로필 및 설정 세트 (Profile, Settings & Session) (v1.3.10)
- **부품 Hooks**: `useHubSession` (Core), `useHub` (Core Logic)
- **부품 Components**: 
  - `HubProfileWidget`, `HubAvatar` (헤더 아바타 위젯)
  - `HubProfileCard` (프로필/닉네임/사진 수정 카드)
  - `HubNotificationCard` (단일 스마트 알림 동의 카드)
  - `HubLogoutCard` (계정 로그아웃 카드)
- **용도**: 게스트/회원 상태 표시 및 페이지 임베딩용 설정 카드 블록 제공.

### ⚙️ 설정 페이지 (/p-settings) 조립 표준
개별 패밀리 앱들은 팝업 모달 대신 온전한 페이지(`/p-settings`)를 생성하여 아래와 같이 SDK 카드 컴포넌트를 조립해 배치해야 합니다.

```tsx
import { 
  HubProfileCard, 
  HubNotificationCard, 
  HubLogoutCard 
} from '@/src/services/merlin-hub-sdk/react';

// 설정 페이지 조립 예시
export default function SettingsPage() {
  const [smartNotification, setSmartNotification] = useState(true);
  
  // 알림 토글 변경 시 개별 앱 백엔드 API 연동
  const handleToggle = async (newValue: boolean) => {
    setSmartNotification(newValue);
    await fetch('/api/subscription/notifications', {
      method: 'PUT',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({ id: userId, enabled: newValue })
    });
  };

  return (
    <div className="space-y-6">
      {/* 1. 상단: 프로필 카드 */}
      <HubProfileCard />

      {/* 2. 중단: 스마트 알림 카드 (앱 전용 설명문 주입) */}
      <HubNotificationCard
        title="알림 설정"
        toggleLabel="🔔 스마트 알림"
        description="이 앱의 서비스 목적에 최적화된 중요 알림 설명을 주입합니다."
        enabled={smartNotification}
        onChange={handleToggle}
      />

      {/* 3. 하단: 로그아웃 카드 */}
      <HubLogoutCard onLogout={() => router.push('/')} />
    </div>
  );
}
```

> 🚨 **[중요] 불필요한 개별 헤더/타이틀 중복 배치 금지**
> `HubProfileCard` 및 `HubNotificationCard` 등 각 카드 블록 내부에는 자체적인 타이틀 영역("프로필 정보", "알림 설정" 등)이 이미 포함되어 있습니다.
> **따라서 카드 상단에 별도로 "설정", "프로필 및 알림" 같은 자체 헤더 텍스트나 부연설명글을 중복으로 하드코딩하지 마십시오. 화면이 지저분해지고 UX가 훼손됩니다.**

## 🔒 2. 이메일 인증 통합 모듈 (Auth Suite)
- **부품**: `useHubAuth` (Core), `HubAuthModal` (Custom)
- **용도**: 6자리 OTP 기반 통합 로그인 및 브랜딩 강화.
- **v1.1.0 변경사항**: `requestOTP(email, appId)`를 통해 앱별 맞춤형 이메일 발송(발송자명, 제목)이 지원됩니다.

### 🎨 UI 커스텀 및 디자인 통합 (통일성 엄수) 가이드
허브 생태계 내의 모든 패밀리 앱들은 이메일 인증창(`HubAuthModal`)에 대해 **동일한 사용자 경험(UX)과 디자인(UI)**을 제공해야 합니다.
개별 앱은 모달을 사용할 때 **오직 아래의 2가지 항목만 Props로 커스텀**할 수 있습니다:
1. **상단 1줄 (앱 로고)**: `appLogoUrl` 프롭을 통해 개별 앱의 고유 로고 이미지를 삽입합니다. (가로로 긴 로고라도 모달이 자동으로 `max-w-[200px] md:max-w-[250px]` 및 비율 축소 최적화를 진행하여 깨지지 않게 렌더링합니다.)
2. **상단 2줄 (액션 목적 텍스트)**: `subtitleActionText` 프롭을 통해 "무료 코인 받아 **[분석에]** 사용하세요" 등 개별 앱의 성격에 맞는 텍스트로 치환합니다.

> 🚨 **[강력 권고] 폰트 및 인풋창 스타일링 임의 변경 금지**
> 이메일 입력창의 텍스트 크기(`text-2xl`)와 플레이스홀더 스타일, 6자리 OTP 인증코드 입력칸의 큼직한 텍스트 크기(`text-3xl`), 테두리 두께(`border-[3px]`) 등은 허브 디자인 표준으로 확립된 규격입니다.
> **어느 패밀리 앱이든 이 글자 크기, 폰트 굵기, 색상 등 내부 스타일을 임의로 변경하지 말고 배포된 SDK의 원본 스타일을 그대로 유지하여 사용해야 합니다.**

### 🎁 [Custom] 혜택 알림 모달 (`HubBenefitModal`, v1.4.0)
로그인(이메일 인증)을 하지 않은 비회원 유저가 무료 체험을 완료했을 때, 가입을 유도하기 위해 노출하는 표준 혜택 안내 모달입니다.
- **`useBenefitTrigger` 연동**: 앱 내에서 무료 체험 완료 시점에 SDK의 `markFreeTrialCompleted()`를 호출하면, 1분 지연 후 모달이 노출됩니다. 또한 "다음에 할게요"를 누른 후 재방문 시에는 5분 체류 후 노출되도록 지연 로직이 내장되어 있습니다.
- **커스터마이징 지원**: 공통 혜택(데이터 보존, 멀티 디바이스 연동) 외에 앱 전용 맞춤 혜택 1가지를 주입할 수 있습니다.

```tsx
import { HubBenefitModal, markFreeTrialCompleted } from '@/src/services/merlin-hub-sdk/react';

// 무료 분석 완료 등 이벤트 발생 시 호출
markFreeTrialCompleted();

// 최상위 레이아웃 혹은 전역 컴포넌트에 배치
<HubBenefitModal 
  customBenefitTitle="관심 채널 신뢰도 변동 알림"
  customBenefitDesc="관심 채널 신뢰도 단계(그린/옐로/레드)가 바뀌면 바로 알림"
  customBenefitIcon="🔔"
/>
```

## 💳 3. 결제 및 코인 (Payment Suite)
- **부품**: `useHubPayment` (Core), `HubPaymentTrigger` (Custom), `HubPurchaseWidget` (Custom)
- **용도**: KCP 결제창 호출, 지갑 잔액 조회 및 공통 결제/이동 내역 UI 페이지 제공.

### [Custom] `HubPurchaseWidget` Component (v1.3.0)
유저의 실시간 잔액 표시, 충전/이용내역 탭 전환, 1,000C / 5,000C / 10,000C 상품 카드 렌더링, 결제 수단 선택(신용카드/휴대폰), 결제창 브릿지 연동, 이용 내역 조회 및 페이징 처리까지 모든 결제 관련 UI와 로직을 내장한 올인원 위젯입니다.
- **v1.3.0 변경사항**: 불필요한 잔액 박스를 제거하여 공간을 최적화했으며, 서버 부하 감소를 위해 이용내역의 필터링(유형/앱/기간) 및 페이징 로직을 모두 프론트엔드(SDK) 단에서 자체 전담하도록 구조가 개편되었습니다. 네이버페이 스타일의 직관적인 필터 UI가 내장되어 있습니다.

```tsx
import { HubPurchaseWidget } from '@/src/services/merlin-hub-sdk/react';

// Next.js Page 컴포넌트 내에서 간편 호출
<HubPurchaseWidget 
  appName="어그로필터" 
  redirectUrl="/" 
/>
```

### [Core] `useHubPayment` Hook (v1.3.0)
KCP 결제 준비 및 승인 통신을 중개하는 Hook입니다. `'card'`(신용카드) 결제를 공식 지원하며, `'phone'`(휴대폰결제)은 현재 심사 중입니다. (계좌이체는 정부 권고 공문에 따라 제거되었습니다.)


### 📊 동적 과금 한도 규격화 정책 (Pricing Cap Policy)
각 개별 앱은 Supabase `family_apps` 테이블의 `client_secret` 필드 내 JSON 형태로 동적 과금 및 마진 배수를 제어할 수 있습니다.
- **`margin_multiplier`**: 동적 요금 계산 시 적용할 곱연산 마진 배수
- **`token_to_coin_rate`**: 토큰 사용량을 코인으로 환산하는 비율 (기본값: 1000토큰 = 1C)
- **`min_price`**: 과금 시 최소로 보장할 최저 코인 금액 (기본값: 10C)
- **`max_price`**: 과금 시 최대로 부과할 최대 코인 금액 한도 (지정하지 않을 시 제한 없음)

*예시 (어그로필터):*
```json
{
  "margin_multiplier": 3,
  "token_to_coin_rate": 1000,
  "min_price": 10,
  "max_price": 100
}
```
위 설정에 따라, 아무리 큰 비디오(예: 70,000토큰 이상)를 분석하더라도 **최대 100C**를 넘지 않도록 제한 한도가 자동으로 안전하게 적용됩니다.

### 🔋 엣지 버퍼링 및 합산 정산 아키텍처 (Edge Aggregation & CCTV Audit)
수시로 적은 금액(1~10C)의 과금이 발생하는 앱(예: Vaulter 등)은 허브 DB 부하와 유저 이용내역 오염을 방지하기 위해 **클라이언트가 아닌 엣지 서버(개별 앱의 백엔드)**에서 토큰/크레딧 사용량을 임시로 버퍼링하는 방식을 표준으로 채택합니다.

- **세션의 정의**: 유저의 웹 입장부터 퇴장까지를 하나의 세션으로 간주하며, 장시간 체류 시 최대 1시간 단위로 세션을 분리합니다. (세션 내역이 0C인 경우는 청구하지 않습니다. 단, 테스트 세션은 별도로 분리합니다.)
- **정산 트리거**: 세션 종료 또는 1시간 경과 시, 개별 앱의 백엔드가 버퍼링된 내역을 합산하여 허브의 `chargeDynamic` API를 단 1회만 호출(Batch Sync)합니다.
- **이중 과금 방어 (Idempotency)**: 허브 결제 API는 개별 앱이 전송하는 `request_id`를 멱등성 키(Idempotency Key)로 판단합니다. 따라서 엣지 서버와 허브 간 통신 장애로 재시도하더라도 동일한 `request_id`를 전송하면 절대 중복 과금되지 않습니다.
- **UI 노출 표준**: 앱 마진 구조 노출 방지 및 UI 직관성을 위해 세부 토큰 수치나 횟수를 노출하지 않습니다. 허브에 전송하는 `display_text` 필드에 시간대만을 포함하여 전송합니다. (예: `"14:15 ~ 15:00 AI 사용"`)
- **엣지 저장소 및 CCTV 주체**: 임시 버퍼링 및 추후 CS 분쟁에 대비한 CCTV 상세 로그 보존을 위한 스토리지(예: DB, KV, Blobs 등) 구축은 **개별 앱의 권한이자 책임**입니다.

## 👥 6. 개별 앱 유저 연동 전략 (User Integration)

개별 앱(Individual App)은 고유의 사용자 테이블을 유지하며, 허브의 공식 식별자를 연동 키로 사용합니다.

### 1단계: DB 스키마 설계 표준 (Standard)
신규 패밀리 앱 개발 시 아래와 같은 네이밍 표준을 권장합니다.
- **테이블 명**: **`users`**
- **연동 컬럼**: **`id`** (UUID) — 허브의 `family_users.id` 값을 그대로 저장합니다.
- **참고**: 어그로필터와 같이 이미 구축된 앱은 `t_users`, `f_id` 등 고유의 네이밍을 사용할 수 있으나, 신규 연동 시에는 위 표준을 따릅니다.

### 2단계: 세션 동기화 (Upsert 패턴)
로그인 성공 시 반환되는 `user.id`를 사용하여 로컬 유저 레코드를 생성하거나 업데이트합니다.

```typescript
// 예시: 허브 로그인 완료 후 로컬 DB(users)와 동기화
const { user, isLoggedIn } = useHub();

useEffect(() => {
  if (isLoggedIn && user) {
    // 로컬 API 호출: 
    // 1. users 테이블에서 id === user.id 인 레코드가 있는지 확인
    // 2. 없으면 신규 생성(Insert), 있으면 최신 정보 업데이트(Update)
    fetch('/api/auth/sync-user', {
      method: 'POST',
      body: JSON.stringify({ 
        hub_id: user.id, // 허브의 family_users.id
        email: user.email 
      })
    });
  }
}, [isLoggedIn, user]);
```

### 3단계: 데이터 관리 책임 분할
- **허브(Hub)**: `family_users` (계정), `family_wallet_balances` (통합 잔액).
- **로컬(Local)**: `users` (앱 전용 설정, 포인트, 활동 로그 등).

## 🎁 7. 초대 시스템 (Referral System) 연동

친구 초대 시 **초대자(Inviter)**와 **가입자(Invitee)** 모두에게 보상을 지급하는 시스템입니다.

### 1단계: 초대 링크 생성
초대자의 코드를 URL 파라미터에 포함하여 홍보 링크를 생성합니다.
- **형식**: `https://your-app.com?ref=INVITE_CODE`
- **SDK 위젯**: `HubReferralWidget`을 배치하면 유저가 자신의 코드를 즉시 확인하고 복사할 수 있습니다.

### 2단계: 가입 시 초대 코드 자동 인식
앱의 진입점(Landing 또는 Login 페이지)에서 URL의 `ref` 파라미터를 저장했다가 가입 시 허브로 전달합니다.

```typescript
// 예시: 초대 코드를 로컬 스토리지에 임시 저장
useEffect(() => {
  const urlParams = new URLSearchParams(window.location.search);
  const refCode = urlParams.get('ref');
  if (refCode) {
    localStorage.setItem('pending_invite_code', refCode);
  }
}, []);

// 회원가입/로그인 시 허브 SDK에 전달
const { login } = useHub();
const pendingCode = localStorage.getItem('pending_invite_code');

const handleJoin = (email, otp) => {
  login(email, otp, pendingCode); // 세 번째 인자로 초대 코드 전달
};
```

### 3단계: 보상 정책 (Hub 기본값)
- **초대자(Inviter)**: 친구 가입 후 최초 기능 소모(가불 정산 또는 분석) 완료 시 **500C** 즉시 지급
- **가입자(Invitee)**: 추천 가입에 따른 별도 추가 보상은 없으나, 모든 가입자는 최초 기능 사용 시 기본 웰컴 보상 **500C** 지급 (초대자 보상은 피초대자의 실제 사용이 일어나기 전까지 PENDING으로 안전하게 보호되어 가계정 어뷰징을 자동 방지합니다.)

## 🎁 8. 비회원 선(先) 체험 가불 정산 및 데이터 이관 스펙

유저에게 가입이나 결제를 요구하기 전에 서비스 핵심 기능을 1회 맛보게 하고, 로그인 가입 시 지급되는 웰컴 보상(`500C`)에서 체험 비용을 사후 차감함과 동시에 해당 활동 이력을 진짜 유저 계정(UUID)으로 이관하는 표준 스펙입니다.

- **스토리지 키 표준**:
  - `pending_usage_fee`: 맛보기 시 발생한 가불 코인 금액 (예: `34`)
  - `pending_video_id`: 맛보기로 생성/분석된 로컬 엔티티 고유 ID
- **인증 연동**: `verifyOTP` 호출 시 `pendingUsageFee`와 `pendingVideoId` 인자를 전달하면 허브 백엔드가 원자적 결제를 대행합니다.
- **데이터 이관**: 가입 성공 직후 개별앱 내 `/api/analysis/link-guest`와 같은 엔드포인트를 호출하여 해당 비디오/기록의 `f_user_id`를 임시 ID(`trial_...`)에서 정식 유저 ID(진짜 UUID)로 변경합니다.

> 📘 **상세 스펙 가이드**: 상세한 데이터 흐름 시퀀스 다이어그램 및 개별앱 연동 코드는 [비회원 선체험 가불정산 및 데이터 이관 가이드](../docs/비회원_선체험_가불정산_및_데이터_이관_가이드.md)를 참고해 주시기 바랍니다.

## 🎨 9. UX/UI 템플릿 표준: 사이드 광고 배치 가이드

광고가 필요한 개별 앱(어그로필터 등)을 개발할 때, 데스크탑 환경에서의 사이드 윙 광고(Wing Ads) 배치 규칙은 다음과 같습니다.

- **우측 단일 광고 원칙**: 메인 콘텐츠(왼쪽/중앙) 렌더링에 집중하고, 광고는 반드시 화면 **우측(Right Side)**에 1개만 배치합니다.
- **F-Pattern 시선 이동 존중**: 유저의 시선은 왼쪽 위에서 시작해 우측으로 이동합니다. 왼쪽에 광고를 배치할 경우 시선을 가로막아 피로감 및 어뷰징 사이트(스팸)로 인식될 우려가 크므로, 글로벌 트렌드(유튜브, 트위터 등)에 맞춰 오른쪽 영역을 광고 공간으로 활용합니다.
- **반응형 하단 광고 전환**: 태블릿/모바일 등 화면 폭이 좁아져 우측 사이드 광고가 사라질 때는, 하단 앵커(Sticky Bottom) 또는 콘텐츠 중간에 가로형 배너를 띄우는 방식으로 유연하게 대체합니다.
