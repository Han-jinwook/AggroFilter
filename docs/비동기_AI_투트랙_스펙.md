# ⚡ 제로대기 비동기 AI 투트랙 작전 계획

## 0) 목표
- 사용자가 분석 시작 직후 **기다리는 느낌 없이** 결과 페이지를 보도록 전환한다.
- AI 처리를 2개 트랙으로 분리해, **빠른 1차 노출 + 정밀 2차 완성** 구조로 운영한다.
- 실패 시에도 화면이 멈추지 않도록 **부분 성공 유지 전략**을 기본으로 한다.

---

## 1) 투트랙 운영 원칙

### Track A (정밀 분석)
- 기존 정밀 분석 로직 유지 (`app/api/analysis/request/route.ts`, `lib/gemini.ts` 중심).
- 최종 점수/평가/랭킹/댓글 등 전체 결과를 책임진다.
- 완료 시 `processingStage = completed`로 마감한다.

### Track B (속도 분석)
- 요약/타임라인/썸네일 스포일러를 우선 생성하는 경량 트랙.
- 결과 페이지 초기 체감속도 개선이 목적.
- `speed_ready` 단계 데이터 노출에 집중한다.

### 공통 원칙
- 프롬프트는 문장 재작성 없이 블록 단위 최소 수정만 허용.
- 크레딧 차감은 기존 1회 정책 유지.
- A 실패 시 B 결과는 유지하고, A 영역만 재시도 가능하게 설계.

---

## 2) 1·2·3단계 UX 작전

### 1단계: 결과 페이지 즉시 진입 + 영상 카드 우선 노출
- 분석 시작 직후 홈 대기화면(퀴즈/완료문구) 없이 결과 페이지로 바로 이동.
- 상단 영상 카드/헤더를 먼저 렌더링.

### 2단계: 1~2초 후 요약/스포일러 노출
- 요약 자막/썸네일 스포일러를 지연 노출(약 1.3초).
- 사용자가 결과가 "진행 중"임을 자연스럽게 인지하도록 구성.

### 3단계: completed 수신 후 전체 분석 결과 노출
- `processingStage = completed` 확인 후 ScoreCard 이하 전체 섹션 렌더링.
- 중간에는 "3차 분석 결과 정리 중" 안내를 스포일러 하단에 배치.

---

## 3) 현재 진행상황 (업데이트)
- **1단계**: 홈 퀴즈/대기 문구 제거 및 결과 페이지 즉시 이동 완료. ✅
- **2단계**: 진입 후 1.3초 뒤 요약/스포일러 노출 로직 반영 완료. ✅
- **3단계**: `completed` 수신 후 전체 섹션 노출 및 보호 로직 반영 완료. ✅
